# Taste Sync — Auto-summarize User Taste

Analyze chat history, detect repeated user corrections and preferences, persist them to `workspace/.disclaude/taste.yaml`, and update `workspace/CLAUDE.md` for automatic agent context injection.

## When to Use This Skill

**Use this skill for:**
- Analyzing chat logs to detect repeated user corrections
- Extracting user preferences (code style, interaction habits, tech choices)
- Persisting preferences so the agent remembers them in future sessions
- Reviewing or resetting accumulated taste rules

**Keywords that trigger this skill**: "总结偏好", "记录习惯", "taste", "偏好总结", "我的习惯", "记住我的偏好", "user preference", "taste sync"

## Context Variables

When invoked, you receive:
- **Chat ID**: Feishu chat ID (from "**Chat ID:** xxx")
- **Message ID**: Message ID (from "**Message ID:** xxx")
- **Sender Open ID**: Sender's open ID (from "**Sender Open ID:** xxx")

## Core Concept

In long-term collaboration, users often correct the same issues repeatedly:

| Scenario | Example Corrections |
|----------|---------------------|
| Code style | "Use const/let, not var" "camelCase for functions" |
| Tech preference | "Use TypeScript not JavaScript" "pnpm not npm" |
| Interaction style | "Be concise" "Conclusion first, then analysis" |
| Project conventions | "Tests in __tests__/" "Chinese commit messages" |

This skill detects these patterns and persists them so the agent auto-adheres in future sessions.

## Analysis Process

### Step 1: Read Existing Taste (if any)

```bash
# Check for existing taste file
cat workspace/.disclaude/taste.yaml 2>/dev/null || echo "No existing taste file"
```

If the file exists, load the current taste rules and their correction counts. New detections will **merge** with existing rules (incrementing counts, not replacing).

### Step 2: Read Chat Logs

Read recent chat log files (last 7 days recommended):

```
workspace/logs/
├── oc_chat1/
│   ├── 2026-04-20.md
│   └── 2026-04-21.md
├── oc_chat2/
│   └── 2026-04-22.md
└── ...
```

**Actions:**
1. Use `Glob` to find all log files: `workspace/logs/**/*.md`
2. Filter to recent files (last 7 days)
3. Read each log file with `Read` tool

### Step 3: Detect Taste Signals

Analyze chat history for these **correction patterns**:

#### Pattern 1: Direct Corrections
User explicitly corrects agent output:
```
User: "不对，应该用 const"
User: "改成 camelCase"
User: "不要这样写，用 TypeScript"
```

#### Pattern 2: Repeated Modifications
User repeatedly modifies the same type of content:
```
User: "帮我改成简洁的回复"
User: "太啰嗦了，简洁点"
User: "先说结论"
```

#### Pattern 3: Explicit Preferences
User states a preference:
```
User: "我习惯用 pnpm"
User: "以后都用 TypeScript 写"
User: "commit message 用中文"
```

### Step 4: Categorize and Merge

For each detected taste, categorize into:

| Category | Examples |
|----------|----------|
| `code_style` | Variable naming, formatting, language features |
| `interaction` | Response length, format, ordering |
| `tech_preference` | Language choice, package manager, framework |
| `project_convention` | Directory structure, naming, workflow |

**Merging rules:**
1. If a similar rule already exists → increment `count`, update `last_seen`, add new example
2. If a new rule is detected → add it with `count: 1`
3. Rules with `count >= 2` are considered **confirmed** and included in CLAUDE.md
4. Rules with `count >= 5` are considered **high priority** and should be listed first

### Step 5: Update Taste File

Write the merged taste data to `workspace/.disclaude/taste.yaml`:

1. Ensure `workspace/.disclaude/` directory exists: `mkdir -p workspace/.disclaude`
2. Write the YAML file with all taste rules
3. Preserve existing rules that were not contradicted

### Step 6: Update CLAUDE.md

Update the taste section in `workspace/CLAUDE.md`:

1. Read the current `workspace/CLAUDE.md`
2. If `<!-- TASTE_SYNC_BEGIN -->` exists → replace the section
3. If not → append the section at the end
4. Only include rules with `count >= 2` (confirmed rules)
5. Format as a clean markdown section

## Commands

This skill supports sub-commands via arguments:

### `/taste-sync` (default: analyze)
Full analysis — read logs, detect patterns, update taste file and CLAUDE.md.

### `/taste-sync list`
Display all current taste rules without re-analyzing:
- Read `workspace/.disclaude/taste.yaml`
- Format as a readable table
- Show correction counts and last seen dates

### `/taste-sync reset`
Clear all accumulated taste:
1. Delete `workspace/.disclaude/taste.yaml`
2. Remove the `<!-- TASTE_SYNC_BEGIN -->...<!-- TASTE_SYNC_END -->` section from CLAUDE.md
3. Confirm the reset

### `/taste-sync add <category> <rule>`
Manually add a taste rule:
1. Append to the specified category in `taste.yaml`
2. Set `source: "manual"`, `count: 1`
3. Update CLAUDE.md

## Minimum Detection Threshold

| Pattern Type | Min Corrections to Record | Min to Include in CLAUDE.md |
|--------------|---------------------------|-----------------------------|
| Direct Correction | 1 | 2 |
| Repeated Modification | 2 | 2 |
| Explicit Preference | 1 | 1 |

---

## Merge Logic (Detailed)

When a new correction is detected for an existing rule:

1. **Exact match**: Same or very similar rule text → increment `count`
2. **Category match**: Same category, different specific rule → add as new rule in same category
3. **Contradiction**: New rule contradicts existing → keep the one with higher count, mark the other as `superseded: true`

**Contradiction example:**
- Existing: "Use npm" (count: 2)
- New: "Use pnpm" (count: 3)
- Result: "Use pnpm" wins, "Use npm" marked superseded

## Integration with Other Systems

### daily-chat-review
The daily-chat-review skill can invoke taste-sync as a follow-up action when it detects repetitive correction patterns. Taste-sync is the specialized tool for extracting and persisting preferences.

### Project System (#2223-#2227)
When the Project system is available, taste files will be stored per-project in `workspace/projects/{name}/taste.yaml`. Until then, the global `workspace/.disclaude/taste.yaml` is used.

### CLAUDE.md
The taste section in CLAUDE.md is the primary mechanism for context injection. The agent naturally reads CLAUDE.md from its working directory, so preferences are automatically loaded every session.

## DO NOT

- Create taste rules from one-off or context-specific requests
- Include sensitive information in taste rules
- Delete taste rules without user confirmation
- Overwrite the entire CLAUDE.md — only modify the `TASTE_SYNC_BEGIN...TASTE_SYNC_END` section
- Add rules with `count < 2` to CLAUDE.md (except explicit preferences)
- Assume user intent — only record what is clearly stated or repeatedly corrected
