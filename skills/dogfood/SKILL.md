# Dogfood — 自动体验最新版本

Detect new version changes, simulate user activities as a "new user", and generate structured feedback reports.

## When to Use This Skill

**Trigger this skill when:**
- User wants to test the current version proactively
- A schedule triggers version-change detection
- User mentions: "自测", "dogfood", "自我体验", "版本体验", "体验报告"

## Context Variables

When invoked, you receive:
- **Chat ID**: Feishu chat ID (from "**Chat ID:** xxx")
- **Message ID**: Message ID (from "**Message ID:** xxx")

---

## State Management

State is persisted in `workspace/data/dogfood-state.json`:

```json
{
  "lastTestedVersion": "0.4.0",
  "lastTestedAt": "2026-04-21T10:00:00Z",
  "lastReportSummary": "All skills functional, 1 minor issue found"
}
```

---

## Workflow

### Phase 1: Version Change Detection

1. **Read current version** from `package.json`:
   ```bash
   node -e "console.log(require('./package.json').version)"
   ```

2. **Read previous state** from `workspace/data/dogfood-state.json`:
   ```bash
   cat workspace/data/dogfood-state.json 2>/dev/null || echo '{}'
   ```

3. **Compare versions**:
   - If `currentVersion !== lastTestedVersion` → proceed to Phase 2
   - If versions match → report "No version change since last test" and stop

### Phase 2: Simulate User Activities

Perform the following activities as a simulated "new user" exploring the system:

#### Activity A: Skill Discovery (5 min)

List all available skills and verify they have valid SKILL.md files:

```bash
find skills/ -name "SKILL.md" -exec echo "=== {} ===" \; -exec head -5 {} \;
```

**Check for:**
- All skills have valid YAML frontmatter
- Descriptions are clear and contain trigger keywords
- No broken skill references

#### Activity B: Schedule Health Check (3 min)

Review all schedule files:

```bash
find schedules/ -name "*.md" -exec echo "=== {} ===" \; -exec head -10 {} \;
```

**Check for:**
- All schedules have valid cron expressions
- Frontmatter is well-formed
- No disabled schedules that should be enabled (or vice versa)

#### Activity C: Code Quality Spot Check (5 min)

Run basic quality checks:

```bash
# Check TypeScript compilation
npm run type-check 2>&1 | tail -20

# Check for lint errors
npm run lint 2>&1 | tail -20

# Run tests
npm run test 2>&1 | tail -40
```

**Note**: If any step fails, record the failure details for the report.

#### Activity D: Recent Changes Review (5 min)

Review recent changes since last tested version:

```bash
# Check git log for recent changes
git log --oneline -20

# Check CHANGELOG for recent entries
head -100 CHANGELOG.md
```

**Identify:**
- New features added
- Bug fixes applied
- Breaking changes
- Areas that might need attention

#### Activity E: Configuration Validation (2 min)

Verify configuration files are valid:

```bash
# Check example config exists and is well-formed
head -50 disclaude.config.example.yaml

# Check .env.example
cat .env.example
```

### Phase 3: Generate Feedback Report

Compile results into a structured report:

```markdown
## 🐕 Dogfood Report — v{version}

**Test Date**: {date}
**Previous Version**: {lastTestedVersion or "N/A (first run)"}

### ✅ Activities Performed

| Activity | Status | Notes |
|----------|--------|-------|
| Skill Discovery | ✅/❌ | {notes} |
| Schedule Health | ✅/❌ | {notes} |
| Code Quality | ✅/❌ | {notes} |
| Changes Review | ✅/❌ | {notes} |
| Config Validation | ✅/❌ | {notes} |

### 📋 Version Changes

{Summary of changes since last tested version}

### 🐛 Issues Found

{Numbered list of issues, if any}

### 💡 Suggestions

{Numbered list of improvement suggestions, if any}

### ⭐ Highlights

{Notable positive findings, if any}

### 📊 Overall Assessment

{Brief overall assessment: healthy / needs attention / critical issues}

---
*Report generated by dogfood skill | Disclaude v{version}*
```

### Phase 4: Persist State & Deliver Report

1. **Update state file**:
   ```bash
   mkdir -p workspace/data
   cat > workspace/data/dogfood-state.json << 'EOF'
   {
     "lastTestedVersion": "{version}",
     "lastTestedAt": "{iso_timestamp}",
     "lastReportSummary": "{one_line_summary}"
   }
   EOF
   ```

2. **Send report** using `send_user_feedback` to the current chatId:
   ```
   send_user_feedback({
     chatId: "{chatId}",
     message: "{report_content}"
   })
   ```

---

## Forced Mode

When invoked with argument `--force` or `force`:
- Skip version change detection
- Always perform all activities
- Useful for on-demand testing between releases

---

## Quality Guidelines

### Good Reports:
- Based on actual test results (not assumptions)
- Actionable issues with specific details
- Concise but comprehensive
- Honest about problems found

### Avoid:
- Fabricating test results without actually running checks
- Ignoring real issues found during testing
- Overly vague descriptions ("something seems off")
- Skipping activities due to time constraints

---

## Schedule Configuration

To enable automatic dogfooding, create a schedule file in `schedules/`:

```markdown
---
name: "Dogfood 自测"
cron: "0 10 * * 1"
enabled: false
blocking: true
chatId: "{your_chat_id}"
---

请使用 dogfood skill 执行自我体验流程。

要求：
1. 检测版本是否发生变化
2. 如果版本未变化，报告状态后结束
3. 如果版本变化，执行完整的 dogfooding 流程
4. 生成结构化反馈报告
5. 使用 send_user_feedback 发送报告到当前 chatId
6. 更新 workspace/data/dogfood-state.json 状态文件
```

## DO NOT

- Fabricate test results — always run actual commands
- Skip the version check unless in forced mode
- Send reports without updating the state file
- Test destructive operations (deploy, push, delete)
- Modify source code during testing
- Create new scheduled tasks during execution
