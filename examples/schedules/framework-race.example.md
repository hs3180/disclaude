# Agent Framework 周度赛马报告

每周一 9:00 分析过去一周的聊天记录，对比不同 Agent 模型/Provider 的表现，生成赛马报告。

**关联 Issue**: #1334

## 配置说明

**重要**: 使用前请将 `chatId` 替换为实际的飞书 Chat ID，并将 `enabled` 设为 `true`。

## 执行步骤

### 1. 获取所有聊天日志文件

```bash
ls workspace/logs/**/*.md 2>/dev/null || echo "No log files found"
```

如果 `workspace/logs/` 目录不存在或为空，跳过本次执行。

### 2. 读取聊天日志

使用 `Glob` 工具查找所有日志文件：
```
Glob workspace/logs/**/*.md
```

对于最近 7 天的日志文件，使用 `Read` 工具读取内容。

### 3. 分析聊天记录

对每段可识别的任务/对话进行分析：

#### 3.1 识别模型和 Provider
- 从消息元数据中提取模型名称（如 claude-sonnet-4-20250514、glm-4 等）
- 如果元数据中无模型信息，从上下文线索推断

#### 3.2 任务分类
- `coding`: 代码编写、调试、重构
- `analysis`: 数据分析、研究、调查
- `communication`: 消息撰写、报告生成
- `operation`: CI/CD、部署、基础设施
- `qa`: 测试、代码审查
- `conversation`: 通用问答、讨论

#### 3.3 性能评估
从以下 5 个维度评估每个任务：
1. **响应效率**: 用户消息到 Bot 回复的时间间隔
2. **任务完成度**: 任务是否成功完成（完成/部分完成/失败）
3. **用户满意度**: 正面/中性/负面信号
4. **工具使用效率**: 工具调用次数与任务复杂度的关系
5. **错误率**: 错误、超时、重试的频率

### 5. 发送报告

使用 `send_user_feedback` 将报告发送到配置的 chatId：

```
send_user_feedback({
  content: [报告内容],
  format: "text",
  chatId: [配置的 chatId]
})
```

## 错误处理

1. 如果日志文件读取失败，跳过该文件继续处理其他文件
2. 如果 `send_user_feedback` 失败，记录日志并重试一次
3. 如果过去 7 天没有足够的任务数据进行有意义的比较（少于 5 个任务），发送简要报告说明"数据不足"

## 设计原则

- **零代码侵入**: 不修改任何核心代码，完全基于已有日志数据
- **AI 驱动分析**: 由 LLM 自主解读数据，不使用硬编码排名算法
- **渐进式**: 先基于现有日志数据，未来可考虑增强日志输出
- **灵活可调**: 评估标准和维度可以随时调整，无需发版
