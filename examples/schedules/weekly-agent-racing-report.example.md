# Agent 性能评估周报

每周一 9:00 分析过去一周的聊天记录，评估 Agent 表现质量，生成对比报告。

**关联 Issue**: #1334
**类型**: Agent Framework 赛马 - 定时日志分析方案

## 配置说明

**重要**: 使用前请将 `chatId` 替换为实际的飞书 Chat ID，并将 `enabled` 设为 `true`。

## 设计说明

本定时任务采用**零代码侵入**方案（Issue #1334 最终方案）：
- 不修改任何核心 Agent 代码
- 不新增任何代码模块
- 完全基于现有聊天日志进行 LLM 分析
- 参考实现: `daily-chat-review` skill

## 执行步骤

### 1. 确定分析范围

分析过去 7 天的聊天记录。

```bash
date -u +"%Y-%m-%dT%H:%M:%SZ"
```

### 2. 获取所有聊天记录文件

聊天记录存储在 `workspace/chat-logs/` 目录下，按日期分组。

```bash
find workspace/chat-logs -name "*.md" -type f -mtime -7 2>/dev/null | head -50
```

如果返回为空，说明过去 7 天内没有聊天记录，跳过本次执行。

### 3. 读取聊天记录

使用 `Glob` 工具查找所有日志文件：
```
Glob workspace/chat-logs/**/*.md
```

对于最近 7 天的日志文件，使用 `Read` 工具读取内容。

### 4. 分析 Agent 性能

对每个聊天会话进行多维度评估：

#### 4.1 响应效率

从消息时间戳推算响应速度：

**用户消息**:
```
👤 [2026-04-14T10:30:00.000Z] (msg_xxx)

用户请求内容...
```

**Bot 响应**:
```
🤖 [2026-04-14T10:30:08.000Z] (msg_xxx)

响应内容...
```

TTFR = Bot 响应时间 - 用户消息时间 = 8 秒

统计指标：
- 平均 TTFR
- P50/P90 响应时间
- 超过 60 秒的慢响应

#### 4.2 任务完成度

分析对话轮次和最终结果：
- 用户请求是否被完整解决？
- 用户是否需要重复提问或纠正？
- 对话是否以任务完成结束？

通过以下信号判断：
- ✅ 完成信号: 用户表示满意、话题自然转换、明确结束
- ❌ 未完成信号: 用户重复相似请求、表示不满、对话中断

#### 4.3 用户满意度

从聊天记录中识别满意度信号：

**正面信号** (至少 1 次):
- "谢谢", "很好", "完美", "解决了", "不错", "thanks", "great"
- 用户主动继续对话 (表示信任)

**负面信号** (需要关注):
- "不对", "错了", "改一下", "不是这个", "重新做", "算了"
- 用户多次纠正相同问题
- 用户放弃对话

#### 4.4 工具使用效率

统计 Bot 消息中的工具调用模式：
- 识别包含工具调用结果的消息
- 评估工具调用是否必要
- 检测工具调用失败情况

#### 4.5 错误模式

识别错误相关消息：
- Bot 消息包含: "失败", "错误", "无法", "error", "failed", "retry"
- 对话突然中断
- Agent 重启信号

### 5. 跨会话对比

将各会话的评估结果汇总对比：

| 维度 | 最佳会话 | 最差会话 | 平均水平 | 趋势 |
|------|---------|---------|---------|------|
| 响应速度 | Xs | Ys | Zs | ↑→↓ |
| 任务完成率 | A% | B% | C% | ↑→↓ |
| 用户满意度 | +A/-B | +C/-D | +E/-F | ↑→↓ |
| 错误率 | X% | Y% | Z% | ↑→↓ |

### 6. 独特优势识别

在量化指标之外，识别 Agent 展现的独特能力：

- 主动提供额外信息和建议
- 创造性的问题解决方式
- 跨领域知识整合
- 良好的沟通风格

这些定性优势是"赛马"中无法量化但同样重要的部分。

### 7. 生成报告

按照以下格式生成报告：

```markdown
## 🏁 Agent 性能评估周报

**分析时间**: [当前时间]
**分析范围**: 过去 7 天
**会话数量**: [分析的会话数]
**消息数量**: [总消息数]

---

### 📊 总体表现

| 指标 | 本周 | 趋势 |
|------|------|------|
| 平均 TTFR | Xs | — |
| 任务完成率 | X% | — |
| 用户满意度比 | +X/-Y | — |
| 错误率 | X% | — |

---

### ⚡ 响应效率

- 平均 TTFR: Xs
- 最快响应: Ys (会话 Z)
- 最慢响应: Ws (会话 V, 原因: ...)
- P90: Xs

### ✅ 任务完成度

- 完成率: X% (N 个请求中 M 个一次解决)
- 常见未完成原因:
  - [原因 1]
  - [原因 2]

### 😊 用户满意度

- 正面信号: X 次
- 负面信号: Y 次
- 比率: +X/-Y

### ❌ 错误模式

| 错误类型 | 出现次数 | 恢复率 |
|---------|---------|--------|
| [类型] | N | X% |

### 🌟 独特优势

- [Agent 展现的独特能力]

### 📋 建议

1. **继续保持**: [表现好的方面]
2. **需要改进**: [具体改进建议]
3. **值得关注**: [趋势性变化]
```

### 8. 发送报告

使用 `send_user_feedback` 将报告发送到配置的 chatId：

```
send_user_feedback({
  content: [报告内容],
  format: "text",
  chatId: [配置的 chatId]
})
```

## 错误处理

1. 如果日志文件读取失败，跳过该文件继续处理其他文件
2. 如果 `send_user_feedback` 失败，记录日志并重试一次
3. 如果没有聊天记录，不发送报告
4. 如果分析结果全是正面，仍然发送报告（正面报告也有价值）

## 扩展选项

### 多周趋势分析

可以扩展此任务，将每周结果保存到 `workspace/data/agent-racing-history.json`，实现多周趋势追踪：

```json
{
  "history": [
    {
      "week": "2026-W15",
      "date": "2026-04-14",
      "avgTtfr": 8.5,
      "completionRate": 0.85,
      "satisfactionRatio": "+12/-3",
      "errorRate": 0.05
    }
  ]
}
```

### 按任务类型分组

可以按任务类型（编码、查询、讨论等）分组统计，识别 Agent 在不同任务类型上的表现差异。

## 注意事项

- 聊天记录文件路径: `workspace/chat-logs/{YYYY-MM-DD}/{chatId}.md`
- 用户消息标识: `👤`
- Bot 消息标识: `🤖`
- 时间戳格式: ISO 8601 (如 `2026-04-10T10:30:00.000Z`)
- 连续用户消息只计算第一条的 TTFR
- 忽略少于 3 次交互的短会话
