# Chats Cleanup

清理 `workspace/chats/` 中的孤儿 `.lock` 文件和 `.stale.*` 残留文件。通过 PID 活跃检测判断锁持有者是否存活，仅移除确认无效的锁文件。

## 配置

- **Chat 目录**: `workspace/chats/`
- **执行间隔**: 每 30 分钟
- **每次最多清理**: 50 个文件（`CHAT_MAX_PER_RUN` 环境变量可覆盖）

## 前置依赖

- Node.js 20.12+（用于 `process.kill(pid, 0)` 活跃检测）

> **⚠️ 无外部依赖**: 本 Schedule 不依赖 `lark-cli` 或其他外部工具，仅使用 Node.js 内置功能。

## 职责边界

- ✅ 清理孤儿 `.lock` 文件（对应 `.json` 文件已不存在）
- ✅ 清理过期 `.lock` 文件（锁持有者进程已终止）
- ✅ 清理损坏的 `.lock` 文件（内容无效）
- ✅ 清理 `.stale.*` 残留文件（中断的锁清理遗留）
- ❌ 不处理 `.json` 会话文件（由 `chat-timeout` skill 负责）
- ❌ 不创建或修改会话文件
- ❌ 不发送消息或操作群组

## 执行步骤

```bash
npx tsx schedules/chats-cleanup.ts
```

### Step 1: 扫描候选文件

遍历 `workspace/chats/` 目录，收集：
- 所有 `.lock` 文件
- 所有匹配 `.stale.{pid}` 模式的残留文件

### Step 2: 清理 .lock 文件

对每个 `.lock` 文件：

1. **孤儿检测** — 检查对应 `.json` 文件是否存在，不存在则移除 `.lock`
2. **活跃检测** — 若 `.json` 存在，读取锁文件内容获取持有者 PID
3. **损坏检测** — 无效内容直接移除
4. **过期检测** — 通过 `process.kill(pid, 0)` 检测进程是否存活，已终止则移除
5. **保留活跃锁** — 持有者进程仍存活的锁文件保留不动

### Step 3: 清理 .stale.* 残留

移除所有 `.stale.{pid}` 文件（这些是中断的锁清理操作遗留的临时文件）。

## 清理策略

| 文件类型 | 判断条件 | 操作 |
|----------|----------|------|
| `.lock` (孤儿) | 对应 `.json` 不存在 | 移除 |
| `.lock` (损坏) | 内容无法解析 | 移除 |
| `.lock` (过期) | 持有者进程已终止 | 移除 |
| `.lock` (活跃) | 持有者进程存活 | 保留 |
| `.stale.*` | 任何情况 | 移除 |

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| Chat 目录不存在 | 正常退出（INFO 日志） |
| `.lock` 文件读取失败 | 跳过（可能已被其他进程移除） |
| `.lock` 文件删除失败 | 记录警告，继续处理下一个 |
| `.stale.*` 文件删除失败 | 记录警告，继续处理下一个 |

## 注意事项

1. **安全清理**: 仅移除确认无效的锁文件，活跃锁不会被误删
2. **PID 检测**: 使用 `process.kill(pid, 0)` 检测进程活跃状态，零副作用
3. **无外部依赖**: 不依赖 `lark-cli`，纯 Node.js 内置功能
4. **限流保护**: 每次执行最多清理 `CHAT_MAX_PER_RUN` 个文件（默认 50）
5. **非阻塞**: `blocking: false`，不会阻塞其他 Schedule 的执行
6. **与 lock.ts 一致**: 使用相同的 PID 检测逻辑，确保行为一致
