# Chats Cleanup

清理 `workspace/chats/` 目录中的孤立 `.lock` 文件。当对应的 `.json` 会话文件已被清理但 `.lock` 文件残留时，本 Schedule 负责移除这些孤立文件。同时清理持有进程已死亡的过期 `.lock` 文件。

## 配置

- **Chat 目录**: `workspace/chats/`
- **执行间隔**: 每 30 分钟
- **每次最多处理**: 50 个（`CHAT_MAX_PER_RUN` 环境变量可覆盖）

## 前置依赖

- Node.js 20.12+（用于 `fs.flock` 文件锁，低版本自动降级为无锁模式）

> **⚠️ 平台要求**: 本 Schedule 使用 TypeScript 实现，通过 `tsx` 运行。无需 `lark-cli`（仅做文件清理，不涉及群组操作）。

## 职责边界

- ✅ 移除孤立 `.lock` 文件（对应 `.json` 已不存在）
- ✅ 移除过期 `.lock` 文件（持有进程已死亡）
- ✅ 移除损坏的 `.lock` 文件（无法解析 PID）
- ❌ 不修改 `.json` 会话文件
- ❌ 不创建或解散群组
- ❌ 不发送消息
- ❌ 不处理 pending/active/expired 状态转换

## 执行步骤

```bash
npx tsx schedules/chats-cleanup.ts
```

脚本完整实现了以下逻辑：

### Step 1: 列出 .lock 文件

遍历 `workspace/chats/*.lock`，获取所有锁文件。

### Step 2: 分类处理

对每个 `.lock` 文件：

1. **路径校验** — 确认文件在 `workspace/chats/` 目录内（防止符号链接逃逸）
2. **孤立检测** — 检查对应的 `.json` 文件是否存在
   - 不存在 → 移除孤立 `.lock` 文件
   - 存在 → 继续检查锁状态
3. **过期检测** — 读取 `.lock` 文件内容，解析持有者 PID
   - 内容损坏 → 移除损坏 `.lock` 文件
   - 持有者 PID 是当前进程 → 跳过（不删除自己的锁）
   - 持有者进程已死亡 → 移除过期 `.lock` 文件
   - 持有者进程存活 → 跳过（锁仍在使用中）
4. **限流保护** — 每次执行最多处理 50 个（可通过 `CHAT_MAX_PER_RUN` 覆盖）

## 清理场景

| 场景 | 条件 | 处理方式 |
|------|------|----------|
| 孤立 `.lock` | 对应 `.json` 不存在 | 移除 `.lock` |
| 过期 `.lock` | 持有进程已死亡 | 移除 `.lock` |
| 损坏 `.lock` | 无法解析 PID | 移除 `.lock` |
| 活跃 `.lock` | 持有进程存活 + `.json` 存在 | 跳过 |

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| Chat 目录不存在 | 正常退出（无需清理） |
| `.lock` 文件已被其他进程删除 | 静默跳过 |
| `.lock` 文件删除失败 | 记录警告，继续处理其他文件 |
| 读取 `.lock` 文件失败 | 静默跳过 |

## 注意事项

1. **无外部依赖**: 不需要 `lark-cli`（仅做文件 I/O）
2. **安全**: 路径遍历保护（`realpath` 校验），不会删除目录外的文件
3. **不会误删**: 只删除 `.json` 已不存在的孤立锁，或持有进程已死亡的过期锁
4. **不删除自己的锁**: 跳过当前进程持有的锁文件
5. **不修改其他文件**: 只处理 `workspace/chats/` 目录下的 `.lock` 文件
6. **幂等**: 重复执行不会产生副作用

## 验收标准

- [ ] 能检测并移除孤立 `.lock` 文件（`.json` 已被清理）
- [ ] 能检测并移除过期 `.lock` 文件（持有进程已死亡）
- [ ] 能检测并移除损坏 `.lock` 文件（无法解析 PID）
- [ ] 不会删除活跃进程持有的 `.lock` 文件
- [ ] 不会删除当前进程自己的 `.lock` 文件
