# Chats Cleanup

清理 `workspace/chats/` 目录中的孤立 `.lock` 文件和过期的残留文件。

## 配置

- **Chat 目录**: `workspace/chats/`
- **执行间隔**: 每 30 分钟
- **锁文件最大存活时间**: 1 小时（超过此时间的 `.lock` 文件被视为孤立文件）

## 前置依赖

- Node.js 20.12+（用于文件操作）

> **⚠️ 平台要求**: 本 Schedule 使用 TypeScript 实现，通过 `tsx` 运行。无需外部依赖（如 `lark-cli`）。

## 职责边界

- ✅ 清理孤立的 `.lock` 文件（对应 `.json` 已不存在）
- ✅ 清理超龄 `.lock` 文件（超过最大存活时间）
- ❌ 不处理 `.json` 文件（由 `chat-timeout` schedule 负责）
- ❌ 不创建或解散群组
- ❌ 不发送消息到群组
- ❌ 不激活 pending 群聊

## 执行步骤

```bash
npx tsx schedules/chats-cleanup.ts
```

脚本完整实现了以下逻辑：

### Step 1: 扫描 `.lock` 文件

遍历 `workspace/chats/*.lock`，查找所有 `.lock` 文件。

### Step 2: 检测孤立 `.lock` 文件

对每个 `.lock` 文件：
1. **检查对应 `.json` 是否存在** — 如果对应的 `.json` 文件已不存在，该 `.lock` 是孤立的
2. **检查 `.lock` 文件年龄** — 如果 `.lock` 文件修改时间超过最大存活时间（1 小时），即使 `.json` 仍存在也视为可清理（可能是死锁）
3. **安全删除** — 只删除符合条件的 `.lock` 文件

### Step 3: 输出清理报告

输出本次清理的统计信息。

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| Chat 目录不存在 | 输出 INFO 日志，正常退出 |
| `.lock` 文件删除失败 | 记录警告，继续处理其他文件 |
| 权限不足 | 记录错误，跳过该文件 |

## 注意事项

1. **安全性**: 只删除 `.lock` 文件，不触碰 `.json` 数据文件
2. **幂等性**: 重复执行不会产生副作用
3. **无外部依赖**: 不需要 `lark-cli`，纯文件系统操作
4. **保守策略**: 只清理明确孤立的 `.lock`（对应 `.json` 已删除）或超龄的 `.lock`（可能死锁）
5. **不创建新 Schedule**: 这是定时任务执行环境的规则
6. **不修改其他文件**: 只处理 `workspace/chats/` 目录下的 `.lock` 文件

## 验收标准

- [ ] 能清理超过最大存活时间的 `.lock` 文件

- [ ] Chat 目录不存在时不会报错
- [ ] `.lock` 文件删除失败时不影响其他文件处理
